## Why

The existing compliance rods (`pseudonymize`, `encrypt`, `validate`) are low-level primitives. Every panel author must manually compose them, wire `@dp`/`@access` decorators, and hope they got GDPR right. There is no reusable, pre-certified unit that encapsulates "this data flow handles personal data correctly under privacy law X." A certified composite `private-data` rod — shipped as a standard rod in core, not hub — would let authors declare privacy intent once and get correct-by-construction compliance, with the certification travelling with the rod.

The rod must be privacy-law agnostic at its base, with specializations for GDPR and then narrower jurisdictions (e.g., German BDSG). This mirrors the `DataSource` union pattern: `private-data { framework: gdpr }` vs `private-data { framework: gdpr.bdsg }`.

## What Changes

- **Introduce standard rods** as a new rod category between basic (18 primitives) and hub (community). Standard rods ship with core, are certified by the project, and compose basic rods.
- **Add `private-data` standard rod** — privacy-law agnostic composite that orchestrates pseudonymize, encrypt, guard, and validate based on a `PrivacyFramework` config.
- **Add `PrivacyFramework` union type** — extensible type tree: `gdpr` (EU), `gdpr.bdsg` (Germany), with room for future frameworks (CCPA, LGPD, POPIA).
- **Define GDPR specialization** — lawful basis (Art. 6), data protection by design (Art. 25), records of processing (Art. 30) entries in manifest, DPIA references (Art. 35).
- **Define BDSG specialization** — employee data rules (Section 26), works council (Betriebsrat) consent tracking, stricter pseudonymization defaults.
- **Add `@privacy` decorator** — panel-level decorator that declares the privacy framework in force, enabling compile-time validation that all personal data flows use appropriate `private-data` rods.
- **Add `PrivateData<T>` generic wrapper type** — marks data as personal data at the type level, carrying field classifications and processing metadata with the data itself. Enables compile-time enforcement: `PrivateData<T>` can only flow through panels with a `private-data` rod or `@privacy` decorator.
- **Add standard personal data models** — pre-classified, ready-to-use `@type` definitions for common personal data patterns: `PersonalContact`, `PostalAddress`, `PersonName`, `UserIdentity`, `EmployeeRecord`, `FinancialAccount`. Each comes with `FieldClassification` baked in, so authors skip manual field tagging.

## Capabilities

### New Capabilities

- `standard-rod-category`: Introduce the standard rod concept — certified composites that ship with core, defined by spec, implemented in core (not hub). Includes the discovery mechanism, extension/specialization model, and how standard rods expand to basic rods during compilation.
- `private-data-rod`: The `private-data` standard rod definition — knots (cfg, arg, in, out, err), privacy-law-agnostic behavior, field classification, purpose limitation, retention, and composition rules for basic compliance rods.
- `privacy-framework-types`: The `PrivacyFramework` union type tree (`gdpr`, `gdpr.bdsg`) and associated config types (`GdprFramework`, `BdsgFramework`, `FieldClassification`, `RetentionPolicy`, `DataSubjectCategory`).
- `gdpr-specialization`: GDPR-specific behavior of the `private-data` rod — lawful basis enforcement, Art. 25 defaults, Art. 30 manifest entries, DPIA references, data subject rights hooks.
- `bdsg-specialization`: German BDSG extensions — Section 26 employee data rules, Betriebsrat consent, stricter pseudonymization for employment context.
- `private-data-model`: The `PrivateData<T>` generic wrapper type — marks any record as personal data with embedded field classifications and processing metadata. Compile-time rule: data typed as `PrivateData<T>` must flow through a `private-data` rod.
- `standard-data-models`: Pre-classified personal data types shipped with core — `PersonalContact`, `PostalAddress`, `PersonName`, `UserIdentity`, `EmployeeRecord`, `FinancialAccount`. Each type carries its `FieldClassification` so the `private-data` rod can derive pseudonymization scope, encryption requirements, and Art. 30 categories automatically.

### Modified Capabilities

- `rod-overview`: The rod taxonomy overview gains a "Standard Rods" section explaining the category and listing `private-data` as the first standard rod.
- `manifest`: Manifest output gains a `privacyRecords` field for Art. 30 record-of-processing entries generated by `private-data` rods with GDPR framework.

## Impact

- **openstrux-spec**: New rod definition in `specs/modules/rods/standard/`, new types in type-system, updated rod overview, updated manifest spec. New conformance fixtures.
- **openstrux-core**: New `packages/generator/src/adapters/nextjs/rods/standard/` folder. Parser must recognize standard rods. Validator must enforce framework-specific constraints. Manifest emitter must produce privacy records.
- **openstrux (hub repo)**: Updated syntax-reference.md to document standard rods. New benchmark cases for certification.
- **Downstream users**: Non-breaking — standard rods are additive. Existing panels using raw `pseudonymize`/`encrypt` continue to work.
