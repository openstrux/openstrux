## Context

`@type` records are the domain model layer of Openstrux. They already map to TypeScript interfaces and Prisma model scaffolds, but the scaffolds are structurally incomplete: no primary key, no foreign key relations, no default values, no index declarations. The gap-fill required to produce a usable `prisma/schema.prisma` negates the strux-first workflow benefit for the most common implementation task.

ADR-020 (Proposed, 2026-03-21) identifies the constructs needed for brownfield adoption (`@external`, `@opaque`, `@pk`, `@table`, `@column`) but has not been implemented. The desired-state generation model in ADR-020 is the right foundation for the output path strategy.

Affected repos:
- **openstrux-spec**: grammar, type-system, syntax-reference (language additions)
- **openstrux-core**: parser, AST, validator (new node types)
- **openstrux** (CLI bundled as `strux.mjs`): generator adapter, config reader, file write step
- **openstrux** (benchmarks): `generate.ts` gap-fill instructions

---

## Goals / Non-Goals

**Goals:**
- Complete `prisma/schema.prisma` generation from `.strux` alone — no gap-fill for owned-type schemas.
- Field-level annotations covering the most common Prisma field attributes: `@pk`, `@default`, `@unique`, `@relation`, `@updatedAt`, `@column`, `@ignore`.
- Block-level annotations: `@@index`, `@@unique`, `@@table`.
- `@timestamps` shorthand to eliminate boilerplate `createdAt`/`updatedAt` declarations.
- `@external` type modifier for brownfield / cross-service table references.
- `@opaque` escape hatch for DB features not yet modelled.
- Desired-state output path: `prisma/schema.prisma` at project root, with overwrite guard.
- Implement and close ADR-020.

**Non-Goals:**
- `strux introspect` command (ADR-020 §1) — generates `.strux` from an existing DB. Valuable but independent; tracked separately.
- Migration execution — OpenStrux declares desired state, Prisma Migrate diffs it. OpenStrux never runs `prisma migrate dev`.
- Bi-directional schema sync.
- ORM targets other than Prisma (Drizzle, TypeORM) — left for future target adapters.
- Composite primary keys (`@@id`) — rare, deferred to a follow-up.
- `@db.XXX` Prisma native-type annotations — too ORM-specific for this abstraction layer.

## Decisions

### D1 — Annotation syntax: inline field decorators, not a separate `@persist` block

**Chosen:** annotations inline after the field type, Prisma-inspired syntax (`@pk`, `@relation(...)`, `@default(now)`).

```
@type Proposal @timestamps {
  id:         string    @pk
  title:      string
  authorId:   string
  author:     Applicant @relation(field: authorId, ref: Applicant.id, onDelete: Cascade)
  status:     ReviewStatus  @default("pending")
  email:      string    @unique
  @@index([authorId, status])
  @@table("proposals")
}
```

**Alternatives considered:**
- **Separate `@persist` block inside `@type`**: `@persist { id: @pk, author: @relation(...) }`. Keeps type declaration and persistence concerns separated, but forces declaration duplication (field name appears twice). Rejected — higher token cost for authors with no semantic gain.
- **`@type` + separate `@model` declaration**: define types separately from persistence models. Cleaner separation of concerns but adds boilerplate for the 90% case where type == model. Rejected — over-engineered for current scope.

### D2 — Naming: `@pk` not `@id` (align with ADR-020)

**Chosen:** `@pk` for the primary key annotation, consistent with ADR-020's proposed naming.

**Rationale:** `@id` is a Prisma-specific term. `@pk` is ORM-agnostic — the Prisma adapter translates it to `@id`; a future SQLAlchemy adapter would translate it to `primary_key=True`. Openstrux is multi-target by design.

**Prisma output:** `@pk` → `@id @default(cuid())` (string), `@id @default(autoincrement())` (int), or explicit via `@pk(default: uuid|cuid|autoincrement)`.

### D3 — Brownfield: desired-state declaration + overwrite guard

**Chosen:** the generator always emits the full desired-state schema for owned types. Brownfield is handled by:
1. `@external type` to exclude non-owned tables from DDL emission.
2. A header sentinel (`// generated by openstrux-generator — do not edit`) on the output file. If the file exists and does NOT have this header, `strux build` warns and skips the write (protecting hand-edited schemas). `--overwrite-schema` forces the write.

**Rationale:** building an incremental schema merger inside the generator duplicates Prisma Migrate (which does this correctly, with DB state awareness). OpenStrux's job is to produce the target state; migration tooling handles the transition. This is consistent with ADR-020 §3.

**Alternative considered:** Parse the existing schema and merge in new models without touching existing blocks. Fragile — Prisma schema has non-trivial syntax, and merging two independent parsers introduces bugs and drift. Rejected.

### D4 — `@timestamps` as a type-level decorator

**Chosen:** `@timestamps` on the `@type` declaration line auto-injects `createdAt: date @default(now)` and `updatedAt: date @updatedAt` into the type body before code generation (AST transformation, not textual).

**Rationale:** 95% of domain types need these two fields. Declaring them explicitly is pure boilerplate. The decorator approach keeps the field count visible in the type body and avoids hidden fields that surprise readers.

**Alternative considered:** A global `@ops { timestamps: true }` in `strux.context`. Too coarse — not every type needs timestamps, and context-level defaults are harder to reason about.

### D5 — `@external` excludes DDL but remains in the type system

**Chosen:** `@external type` participates in the type checker (relations can reference it, panels can use it as input/output types, Zod schemas can be generated for it) but the generator emits no Prisma model block for it. This matches ADR-020 §2.

**Rationale:** The external type still needs to be validated at intake (Zod), and relations from owned types to external types need to appear in the Prisma schema as relation fields (without a `model` block for the external side). Prisma handles this correctly when the external model is managed outside the schema or via an existing DB.

### D6 — `@opaque` preserved as a comment in Prisma output

**Chosen:** `@opaque` statements are emitted as `// @opaque <content>` comments in the Prisma model block. They do not affect generated TypeScript.

**Rationale:** Preserves round-trip fidelity (introspect → build → introspect produces the same opaque annotations) without requiring the Prisma adapter to understand DB-specific syntax like partial index predicates or trigger names.

### D7 — Output path: `output.prisma_schema` in `strux.config.yaml`

**Chosen:** Default is `prisma/schema.prisma` (project root). Overridable via:

```yaml
output:
  dir: ".openstrux/build"
  prisma_schema: "prisma/schema.prisma"   # default; override to change path
```

**Rationale:** Most projects have a single `prisma/` directory at the root. Making it configurable handles monorepos with non-standard layouts without special-casing.

## Risks / Trade-offs

**[Risk] `@pk` naming diverges from Prisma convention** → Mitigation: document the mapping clearly in syntax-reference and error messages. The `strux build --explain` output shows the Prisma expansion for each annotation.

**[Risk] Overwrite guard breaks CI on fresh clones** → `prisma/schema.prisma` does not exist on a fresh clone, so the guard only triggers when the file exists without the generated header. Fresh-clone CI builds always succeed. If teams commit a hand-edited schema and then add `.strux` annotations, they must run `strux build --overwrite-schema` once.

**[Risk] `@relation` back-references (the array side) are not declared in `.strux`** → Mitigation: the adapter emits the Prisma relation array field automatically on the referenced type's model block (e.g. `posts Post[]` on `User`). The `.strux` author only declares the owned (FK) side; the inverse is inferred. This matches Prisma's model and keeps `.strux` types lean.

**[Risk] `@external` + `@relation` produces incomplete Prisma schemas** → If an owned type has a `@relation` to an `@external` type, the Prisma schema has an orphaned relation. Mitigation: validator emits a `W_EXTERNAL_RELATION` warning; the generated schema comment notes the external model is expected to exist in the DB. This is standard Prisma practice for multi-schema setups.

**[Trade-off] `@timestamps` hides fields** → Engineers reading the type see fewer fields than are in the DB. Mitigation: `strux build --explain` prints the expanded type with injected fields. The schema.prisma file always shows the full list.

## Migration Plan

1. **openstrux-spec**: add grammar productions and update syntax-reference, type-system. These are additive — existing `.strux` files parse and validate unchanged.
2. **openstrux-core**: add AST nodes, update parser and validator. Ship as a minor version bump.
3. **CLI (strux.mjs)**: update generator; add `--overwrite-schema` flag; update config reader. Deploy bundled with the next CLI release.
4. **Benchmark**: update `generate.ts` gap-fill list and prompt template. Takes effect immediately for new benchmark runs.
5. **ADR-020**: update status to Accepted.

No database migrations required. No breaking changes to existing `.strux` files. One breaking change to file output paths (`.openstrux/build/prisma/schema.prisma` → `prisma/schema.prisma`) — documented in release notes and migration guide.

**Rollback**: revert CLI and spec; the old output path behaviour is trivially restorable via config.

## Open Questions

- **OQ-1**: Should `strux introspect` be included in this change or tracked separately? Current decision: separately — introspect is a substantial CLI addition and is not required to close the gap-fill problem.
- **OQ-2**: Should `@@id([f1, f2])` composite primary keys be in scope? Current decision: deferred — uncommon in grant-workflow domain, adds significant validator complexity.
- **OQ-3**: Should `@relation` on the owned side auto-emit the inverse array field on the referenced type, or should authors declare it explicitly? Current decision: auto-emit the inverse — keeps `.strux` types lean and matches Prisma's convention.
