## Why

`@type` records today generate Prisma model scaffolds that are structurally incomplete: no `@pk`, no `@relation`, no `@default`, no `@updatedAt`. The generated `.openstrux/build/prisma/schema.prisma` is unusable by Prisma CLI and must be fully rewritten as a manual gap-fill — meaning every field the author already expressed in `.strux` gets re-derived from scratch in a separate file. This defeats the strux-first workflow and introduces a drift vector between the type definition and the schema.

Separately, ADR-020 (Proposed) identifies brownfield adoption as a blocker but has not yet been implemented. The two concerns are inseparable: completing field-level annotations without a brownfield strategy produces a generator that overwrites production schemas. This change implements both together.

## What Changes

### Language additions (field-level annotations on `@type` records)

**Field-level decorators** (inline after the field type):

| Annotation | Meaning | Prisma output |
|---|---|---|
| `@pk` | Primary key; default generator inferred from field type (`cuid` for string, `autoincrement` for int) | `@id @default(cuid())` |
| `@pk(default: uuid)` | Primary key with explicit generator | `@id @default(uuid())` |
| `@pk(default: autoincrement)` | Integer PK | `@id @default(autoincrement())` |
| `@default(now)` | Timestamp default | `@default(now())` |
| `@default(true\|false\|"val"\|number)` | Scalar default | `@default(value)` |
| `@unique` | Unique constraint | `@unique` |
| `@relation(field: localField, ref: Model.field)` | Foreign key (owned side) | `@relation(fields: [...], references: [...])` |
| `@relation(field: localField, ref: Model.field, onDelete: Cascade\|SetNull\|Restrict\|NoAction)` | FK with referential action | `@relation(..., onDelete: Cascade)` |
| `@updatedAt` | Auto-update timestamp | `@updatedAt` |
| `@column("name")` | Map field to DB column name | `@map("name")` |
| `@ignore` | Exclude from generated client | `@ignore` |

**Block-level statements** (inside `@type` body, prefixed `@@`):

| Annotation | Meaning | Prisma output |
|---|---|---|
| `@@index([f1, f2])` | Composite index | `@@index([f1, f2])` |
| `@@unique([f1, f2])` | Composite unique constraint | `@@unique([f1, f2])` |
| `@@table("name")` | Map type to DB table name | `@@map("name")` |

**Type-level decorators** (on the `@type` declaration line):

| Annotation | Meaning | Effect |
|---|---|---|
| `@timestamps` | Auto-inject `createdAt`/`updatedAt` | Adds `createdAt: date @default(now)` and `updatedAt: date @updatedAt` |

### Brownfield constructs (from ADR-020, implemented here)

- **`@external type`**: marks a type as non-owned. The generator references it in relations but emits no DDL. Satisfies cross-service and legacy table references.
- **`@opaque` block statement**: captures DB features not yet modelled (triggers, partial indexes, custom types). Preserved through parse → IR → emit; not interpreted by the compiler.

### Generator output path and update strategy

- `prisma/schema.prisma` is written to the **project root** (not `.openstrux/build/`), configurable via `output.prisma_schema` in `strux.config.yaml`.
- The generator follows **desired-state declaration**: each `strux build` emits the full current desired schema for all owned (non-`@external`) types. Teams use `prisma migrate diff` (or their chosen migration tool) to compute the incremental delta from current DB state to desired state.
- The generated file is headed `// generated by openstrux-generator — do not edit`. If the file exists and does NOT start with this header, `strux build` prints a warning and aborts the write (protecting hand-edited schemas). Pass `--overwrite-schema` to force.
- **`@external` types** exclude their models from the emitted schema, enabling brownfield adoption without emitting DDL for tables the team does not own.

**BREAKING**: `prisma/schema.prisma` output path changes from `.openstrux/build/prisma/schema.prisma` to `prisma/schema.prisma` (project root).

### Benchmark impact

- `prisma/schema.prisma` is removed from the openstrux-path gap-fill list. Agents annotate types in `.strux` and get a complete schema from `strux build`.
- `prisma/seeds/seed.ts` remains a gap-fill (seed data is not expressible in `.strux`).

## Capabilities

### New Capabilities

- `type-field-annotations`: Field-level (`@pk`, `@default`, `@unique`, `@relation`, `@updatedAt`, `@column`, `@ignore`) and block-level (`@@index`, `@@unique`, `@@table`) persistence annotations on `@type` records, plus `@timestamps` type decorator and `@external` type modifier and `@opaque` escape hatch. Includes syntax, grammar, semantics, validation rules, and Prisma/TypeScript mapping.

### Modified Capabilities

- `generator`: Prisma schema written to project root; desired-state overwrite strategy with `--overwrite-schema` guard; `@external` types excluded from DDL.
- `target-adapter-ts`: `@type` → Prisma model mapping extended with all field annotation translations; `@external` models emitted as relations only; `@opaque` content preserved as comments.

## Impact

- **openstrux-spec** (`specs/core/`): grammar, type-system, syntax-reference — field annotation productions, `@external` modifier, `@opaque` statement.
- **openstrux-core**: parser (new AST nodes: `FieldAnnotation`, `TypeBlockAnnotation`, `OpaqueStatement`), validator (one `@pk` per non-union `@type`, `@relation` ref must resolve to existing `@type`, `@external` types cannot have `@pk`).
- **CLI (`strux.mjs`)**: `emitRecord` extended; config reader for `output.prisma_schema`; write step guarded by header check; `--overwrite-schema` flag.
- **Benchmark (`generate.ts`)**: openstrux path gap-fill list updated; prompt instructions updated.
- **`docs/syntax-reference.md`** (openstrux repo): user-facing copy of the syntax reference updated with persistence annotation section, alias table, and updated examples. This is the primary reference end-users and LLMs consult when writing `.strux` files.
- **`docs/getting-started.md`**: `@type` examples updated to show `@pk`; note added that `strux build` produces a complete `prisma/schema.prisma`.
- **ADR-020**: this change implements ADR-020's `@external`, `@table`, `@column`, `@pk`, `@opaque` constructs. ADR-020 status should be updated to Accepted once this change ships.
- **No changes** to panel/rod syntax, runtime behaviour, or existing `.strux` files — all new annotations are additive and optional.
