# Openstrux v0.6.0 — Release Milestone

**Target date:** 2026-03-27
**Status:** In progress

---

## Objective

First end-to-end working version of Openstrux. Given `.strux` panel files, the toolchain parses, validates, locks, and generates working TypeScript artifacts. Demonstrated on the grant-workflow use case (NLnet MVP phases P0-P2). Fulfills ADR commitments deferred to v0.6: explanation generation (ADR-013), `@cert` enforcement (ADR-011), config inheritance resolution, lock-determinism (ADR-014), policy resolution plumbing (ADR-010).

## Release gate

Per `governance/BRANCHING.md` and `docs/manifesto/MANIFESTO_OBJECTIVES.md`:

- [ ] All 7 manifesto principles score PASS or WARN (no FAIL) — WARN-data (incomplete measurement) is acceptable for alpha; WARN-result (below threshold) requires a remediation plan
- [ ] Benchmark cases B001-B010 executed against 1+ frontier LLM (**alpha-only evidence** — full gate requires 20+ cases, 3+ LLM families; deferred to v0.7.0)
- [ ] Grant-workflow generation comparison (direct-TS vs strux) measured: input/output tokens, time
- [ ] Conformance fixtures: valid/, invalid/, and golden/ populated for grant-workflow P0-P2 constructs
- [ ] TypeScript target generates compilable output for all P0-P2 panels
- [ ] Locked builds are deterministic (same source + same lock = same output)
- [ ] Parser passes valid/invalid fixtures
- [ ] Validator enforces type paths, AccessContext scope narrowing, and E_CERT_IN_CONTEXT
- [ ] Config resolution produces flattened manifests
- [ ] All 18 rod types handled without crashing (Tier 2 stubs excluded from benchmark claims)
- [ ] B001-B010 published with scorecard
- [ ] Grant-workflow starter repo runs end-to-end
- [ ] RFC-0001 (TypeScript target adapter) accepted
- [ ] Generator spec (`specs/modules/target-ts/generator.md`) published
- [ ] `openstrux-spec` tagged first, `openstrux-core` after conformance pass, hub after scorecard published

## Release criteria summary

The minimum credible release: parser passes valid/invalid fixtures; validator enforces type paths, AccessContext scope narrowing, and E_CERT_IN_CONTEXT; config resolution produces flattened manifests; locked builds are deterministic; TypeScript target handles all 18 rods without crashing; B001-B010 exist with published scorecard; and the grant workflow runs end to end.

Full fidelity reverse translation and CST preservation are NOT release blockers — `--explain` plus source locations is the accepted near-term answer for human traceability.

## Out of scope for v0.6.0

- Beam Python target
- Grant-workflow phases P3-P6
- Benchmark cases B011-B030
- Grammar, IR, semantics spec stub sections
- Manifest -> `.strux` decompilation (ADR-013 re-deferred to v0.7.0)
- Cost-based pushdown optimization (ADR-015 — raw-expr in v0.6.0)
- Tiered policy store runtime behavior (ADR-010 — static tracking only)
- Lock file merge/conflict resolution
- Full expression AST parsing (raw-expr only in v0.6.0)

## Recommended future work: Token optimization ADR

A dedicated ADR should be created in `openstrux-spec` formalizing the token-efficient authoring pattern:
- Context cascade: root `strux.context` -> domain `strux.context` -> small delta panels
- Named references over repeated inline config
- Shorthand syntax with implicit chaining
- What to inherit (dp, access defaults, named source/target, ops, sec) vs what to keep local (panel logic, rod arguments, cert declarations)

This ADR would make the pattern normative and ensure the grant-workflow use case follows it. Deferred to implementation phase when concrete token measurements validate the pattern.
