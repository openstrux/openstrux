# Supply Chain Security

This document defines artifact attestation, dependency hygiene, and release verification for Openstrux. Controls are graduated — stricter at release, lighter during development.

---

## GitHub Actions pinning policy

All third-party GitHub Actions must be pinned to a full commit SHA, not a tag or branch:

```yaml
# Correct
- uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683  # v4.2.2

# Wrong — tag can be moved
- uses: actions/checkout@v4
```

**Enforcement:** new workflow files with unpinned actions fail the L0 lint check. Existing workflows must be migrated before v1.0.

**First-party actions** (within the `openstrux` GitHub org) may use branch refs on `main` during active development; they must be pinned before any production release.

## Release outputs

Every release of `openstrux-core` produces three distinct artifacts. They are separate files, with separate roles:

| Artifact | File | Purpose |
|---|---|---|
| **Package** | `openstrux-core-vX.Y.Z.tar.gz` | What was built — the source archive or compiled output |
| **SBOM** | `openstrux-core-vX.Y.Z.sbom.json` | What is inside it — full dependency inventory in CycloneDX JSON |
| **Provenance bundle** | `openstrux-core-vX.Y.Z.tar.gz.bundle` | How it was built — cosign signature + OIDC certificate proving the build identity |

All three are attached to the GitHub Release. Failure to produce any one of them blocks the release pipeline.

This separation makes the path to stronger SLSA provenance straightforward: the package and SBOM are already separate; upgrading SLSA level means strengthening the provenance bundle, not restructuring the release.

---

## SBOM

- Format: CycloneDX JSON
- Generator: `cyclonedx-bom` (Python) and `cyclonedx-node` (TypeScript), run during the release pipeline
- Not required for pre-release (`alpha`, `rc`) builds — generated but not published

---

## Artifact attestation (provenance bundle)

Release packages are signed using cosign keyless signing:

```sh
cosign sign-blob --bundle openstrux-core-vX.Y.Z.tar.gz.bundle openstrux-core-vX.Y.Z.tar.gz
```

- Signing uses GitHub OIDC — no long-lived private keys stored anywhere
- The signing certificate is bound to the GitHub Actions workflow identity (`release.yml`)
- The bundle contains the signature, certificate chain, and transparency log entry

**SLSA target:** SLSA Build L1 for v0.x (provenance generated by the build; not independently verified). SLSA Build L2 for v1.0 (provenance uploaded to Rekor transparency log and independently verifiable).

---

## Release verification

Anyone can verify a release artifact:

```sh
cosign verify-blob \
  --bundle artifact.tar.gz.bundle \
  --certificate-identity-regexp "https://github.com/openstrux/openstrux-core/.github/workflows/release.yml" \
  --certificate-oidc-issuer "https://token.actions.githubusercontent.com" \
  artifact.tar.gz
```

Verification instructions are included in every GitHub Release description.

## Dependency hygiene

### Adding a new dependency

Before adding a dependency to any Openstrux repo:

1. Check license against the allowlist in `governance/AIDEVSECOPS.md` (MIT, Apache-2.0, BSD-2/3, ISC, MPL-2.0).
2. Run `osv-scanner` against the candidate package.
3. Note the dependency in the PR description with: name, version, license, purpose.

No dependency is added silently. Every dependency addition is visible in the PR diff and reviewable.

### Removing a dependency

Prefer removal over workarounds when a dependency becomes a security liability and no upstream fix is available within SLA.

### Transitive dependencies

Transitive dependency vulnerabilities are treated the same as direct ones — triaged and resolved per `governance/SECURITY_OPERATIONS.md`. Lock files (`package-lock.json`, `requirements.lock`, `go.sum`) are committed and kept up to date.

## Graduated controls by stage

| Control | Pre-v1.0 (current) | v1.0+ |
|---|---|---|
| Action pinning | Required for third-party; advisory for first-party | Required for all |
| SBOM | Generated per release, published | Generated per release + pre-release |
| cosign signing | Release artifacts | All artifacts including pre-release |
| SLSA level | L1 | L2 |
| Provenance log | GitHub Actions audit log | Rekor transparency log |
