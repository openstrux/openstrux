# B014 — Certification Token Efficiency

**id:** B014
**category:** token-efficiency
**constructs:** Art. 30 record generation, metadata queryability
**step:** 2 (requires step 1 backend generation)

## Intent Prompt

Produce a GDPR Article 30 Record of Processing Activities for the generated system (same task as B011). This case measures the token cost of the certification step itself — how many tokens the LLM consumes (input + output) to produce the record.

## Expected Validation

Same as B011 — output must be valid Art. 30 JSON with both processing activities populated.

## Expected Metrics

| Metric              | Expected (openstrux) | Expected (direct)  |
|---------------------|---------------------|-------------------|
| inputTokens         | lower               | higher            |
| outputTokens        | comparable          | comparable        |
| completenessScore   | ≥ 80                | ≥ 60              |
| tokenizer           | cl100k_base         | cl100k_base       |

**Hypothesis:** The openstrux path consumes fewer input tokens for the certification step because the LLM can read compact `.strux` metadata rather than scanning verbose TypeScript source files to discover data flows and PII fields. The queryable structure reduces the search space.

## Notes

Input token count is measured from the LLM API call for step 2. Results are recorded in `step-2/generation-meta.json` alongside `step-2/scoring.json`.
