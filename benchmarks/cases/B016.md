# B016 — Meaningful Code Surface Token Size

**id:** B016
**category:** token-efficiency
**constructs:** code surface area, structural authoritativeness of `.strux` vs generated TypeScript
**step:** 1 (measured after backend generation, before certification)

## Intent Prompt

This case has no LLM prompt — it is a measurement case. After step 1 generation completes, the benchmark runner counts the tokens in the code an LLM would need to read to understand the system well enough to make a change.

## Measurement Definition

**Openstrux path:** token count of all `.strux` source files in the worktree.
- Includes: `**/*.strux` (everywhere under the repo root)
- Excludes: `.openstrux/build/**` (TypeScript emitted by `strux build`), `node_modules/**`

**Direct path:** token count of all TypeScript source files under `src/`.
- Includes: `src/**/*.ts`
- Excludes: `*.d.ts` declaration files, `node_modules/**`, `dist/**`, `.next/**`

Token count uses cl100k_base encoding (tiktoken).
Result is written to `code-surface.json` in the step 1 result directory.

## Expected Metrics

| Metric          | Expected (openstrux)  | Expected (direct)     |
|-----------------|-----------------------|-----------------------|
| tokenCount      | lower                 | higher                |
| fileCount       | ≤ 5                   | ≥ 8                   |
| tokenizer       | cl100k_base           | cl100k_base           |

**Structural compression hypothesis:** `.strux` files are the authoritative source of truth and contain only declarative structure — no boilerplate, no type annotations, no imports. The direct path's TypeScript source is more verbose for the same domain model, so the token surface is larger.

## Output Format (`code-surface.json`)

```json
{
  "path": "openstrux | direct",
  "tokenCount": 4200,
  "fileCount": 7,
  "files": ["src/data/submission.strux", "..."],
  "excludedPatterns": ["**/.openstrux/build/**", "..."],
  "tokenizer": "cl100k_base"
}
```

## Notes

This metric is a property of the generated artifact, not of the LLM call. It measures the *reading burden* imposed on any LLM tasked with making a subsequent change — directly predictive of step 3 (propagation) difficulty.
