# B012 — Privacy Field Classification Accuracy

**id:** B012
**category:** certification-and-audit
**constructs:** field classification, data category identification, sensitivity levels
**step:** 2 (requires step 1 backend generation)

## Intent Prompt

List all personal data fields processed by the generated system. For each field, classify it by:
- Data category (identifying, quasi-identifying, sensitive, non-personal)
- Sensitivity level (high, medium, low)
- Processing purpose(s)
- Whether it is pseudonymized, encrypted, or stored in plaintext

Output as a JSON array with one entry per field.

## Expected Validation

- All known PII fields identified: applicant name, email, organization name, proposal content
- Correct categories assigned (name/email = identifying; org = quasi-identifying; proposal = potentially sensitive)
- Pseudonymization/encryption status matches actual implementation (from blinded packet logic or `pseudonymize` rod)
- No non-personal fields incorrectly classified as PII

## Expected Metrics

| Metric               | Expected (openstrux) | Expected (direct) |
|----------------------|---------------------|-------------------|
| fieldCoverageScore   | ≥ 90                | ≥ 75              |
| classificationScore  | ≥ 80                | ≥ 65              |
| queryabilityScore    | 80–100              | 30–50             |

## Openstrux Path Advantage

The openstrux path reads field classifications directly from `@type` annotations and field-level metadata. The direct path must infer classification from variable names, Prisma field types, and surrounding code logic — a fragile heuristic approach.
