# B011 — Art. 30 Record from Generated Backend

**id:** B011
**category:** certification-and-audit
**constructs:** `@dp`, `@access`, field classification, data flow tracing
**step:** 2 (requires step 1 backend generation)

## Intent Prompt

After generating the grant workflow backend, produce a GDPR Article 30 Record of Processing Activities for the system. For each data flow that processes personal data, state:

1. Controller identity and DPO
2. Processing purpose
3. Lawful basis
4. Categories of data subjects
5. Categories of personal data processed
6. Recipients or categories of recipients
7. Retention period
8. Technical and organizational security measures
9. DPIA reference (if applicable)

Output as structured JSON conforming to `benchmarks/schemas/art30-record.json`.

## Expected Validation

- Output is valid JSON conforming to the Art. 30 schema
- At least two processing activities present (intake + eligibility)
- All 8 required fields populated for each activity
- Personal data categories include at least: name, email, and one identifier
- Technical measures include pseudonymization or equivalent privacy control

## Openstrux Path Advantage

The openstrux path can derive the Art. 30 record structurally from:
- `@dp` blocks on panels (`record: true`, `basis: "contract"`)
- `@access` blocks (`purpose`, `operation`)
- `@type` field definitions (classify PII by field type and annotations)
- Manifest `privacyRecords` (when `private-data` rod is available)

The direct path must trace data flows by reading TypeScript source, Prisma schema, and comments.
