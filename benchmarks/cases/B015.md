# B015 — Technical Measures Identification

**id:** B015
**category:** certification-and-audit
**constructs:** pseudonymization detection, encryption detection, access control identification
**step:** 2 (requires step 1 backend generation)

## Intent Prompt

Identify all technical and organizational security measures applied to personal data in the generated system. For each measure, state:
- Measure type (pseudonymization, encryption, access control, audit logging, etc.)
- Which data or data flows it applies to
- How it is implemented (code reference or structural reference)
- GDPR relevance (which principle it satisfies)

Output as a JSON array with one entry per measure, then include the `technicalMeasures` section of the Art. 30 record.

## Expected Validation

- Pseudonymization identified (blinded packet stripping applicant identity)
- Access control identified (role-based middleware on reviewer routes)
- Audit logging identified (AuditEvent records)
- Each measure includes a reference to its implementation (file path or `.strux` rod)
- At least one measure mapped to a GDPR principle (Art. 5 or Art. 25)

## Expected Metrics

| Metric                   | Expected (openstrux) | Expected (direct) |

| measuresIdentified       | ≥ 3                 | ≥ 3               |

## Openstrux Path Advantage

The openstrux path reads technical measures directly from compliance rods (`pseudonymize`, `encrypt`, `guard`) in `.strux` source. The direct path must locate privacy logic scattered across service files, middleware, and Prisma hooks.
