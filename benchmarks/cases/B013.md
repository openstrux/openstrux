# B013 — Change Propagation Correctness

**id:** B013
**category:** certification-and-audit
**constructs:** change propagation, re-certification, structural update
**step:** 3 (requires step 2 certification)

## Intent Prompt

Add `phone_number` as a new PII field to the grant application intake. It should be:
- Stored on the applicant identity record
- Classified as identifying data (sensitivity: high)
- Included in the blinded packet pseudonymization
- Reflected in the updated Art. 30 record (add to personalDataCategories for the intake activity)

After making the change, produce an updated Art. 30 record reflecting the new field.

## Expected Validation

- `phone_number` added to the data model (Prisma schema or `.strux` `@type`)
- `phone_number` included in pseudonymization logic (BlindedPacket or `pseudonymize` rod)
- Updated Art. 30 JSON includes `phone_number` in `personalDataCategories` for the intake activity
- No regressions in existing unit tests

## Expected Metrics

| Metric                    | Expected (openstrux) | Expected (direct) |
|---------------------------|---------------------|-------------------|
| filesModified             | ≤ 3                 | ≥ 5               |
| art30UpdateCorrect        | true                | true              |
| pseudonymizationUpdated   | true                | true              |
| regressionTests           | pass                | pass              |

## Openstrux Path Advantage

The openstrux path adds the field once in the `@type` definition and optionally marks it with a classification. `strux build` regenerates the Prisma schema and TypeScript types automatically. The Art. 30 record update is a single-field addition. The direct path must find and update every file that touches the field: Prisma schema, TypeScript types, service layer, route handler, and the Art. 30 record manually.
